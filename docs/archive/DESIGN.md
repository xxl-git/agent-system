# 🏗️ Agent System — 设计文档

> **创建时间**：2026-06-16 22:47
> **角色**：AI 为主架构师，用户为补充
> **核心理念**：主动性、自我进化、模型自适应


## 一、愿景

构建一个 **主动型 Agent 操作系统**，而非被动应答的聊天机器人。核心差异：

| | OpenClaw（被动） | 本系统（主动） |
|---|---|---|
| 触发方式 | 等用户消息 | 可主动规划、分解、发起任务 |
| 技能获取 | 人工安装 | 自动感知缺口→提交申请→审核→开发→装备 |
| 记忆 | 文件为主 | 文件+数据库双轨，语义检索 |
| 模型交互 | 固定协议 | 自动探测模型习惯，调整交互方式（磨合期） |
| 协作 | 单 Agent | 多 Agent 协作 |
| 任务路由 | 人工选模型 | 难度评估→自动路由（本地/云端），有兜底 |

## 三、模块详细设计

### 3.0 主动性引擎 (Proactivity Engine) ⭐ 核心差异化

**两种模式，无需模型推理即可触发：**

#### A 型 — 空闲自驱 (Idle Self-Tasking)

```
心跳定时器（每 N 分钟）
  ↓
检查待办队列：
  ├── 技能申请待审核？ → 启动审核流程
  ├── 记忆待整理？ → 启动记忆摘要
  ├── 项目有更新？ → git status / 文件变更检测
  ├── 外部事件？ → 邮件/日历/天气检查
  └── 无待办 → 安静休眠
```

**设计要点**：
- 心跳频率可配置，默认 5 分钟
- 每次心跳执行不超过 N 秒（防阻塞）
- 发现任务后生成优先级排序
- 非紧急任务排队，不打扰用户

#### B 型 — 任务主动分解与追问 (Active Decomposition & Clarification)

```
用户："帮我做一个媒体中心"

B 型自动行为：
  1. 意图不清晰时主动追问：
     "你要桌面应用还是 Web？有什么特别的功能需求？"
  
  2. 自动分解为子任务 DAG（见 3.1 调度器）
  
  3. 执行前预检：
     - 检查所需技能是否存在
     - 检查依赖是否满足（ffmpeg 装了没？）
     - 缺口发现 → 触发技能申请
  
  4. 执行中主动汇报：
     "数据库设计完成，正在写 UI..."
     "遇到问题：端口 9700 被占用，已自动换到 9701"
  
  5. 完成后主动总结：
     "媒体中心开发完成。文件在 D:\xxx。启动了 3 个子 Agent，
      申请了 1 个新技能(ffmpeg)，耗时 8 分钟。"
```

**设计要点**：
- 追问阈值：信息不足时自动触发（不依赖模型判断，规则驱动）
- 分解粒度：子任务不超过预估 5 分钟/个
- 预检清单：依赖、权限、磁盘、网络
- 进度回调：每个子任务完成时汇报

### 3.1 任务调度器 (Task Orchestrator) ⭐ 核心

**职责**：大脑。接收意图，分解为子任务，分配 Agent 执行，追踪进度。

```
用户："帮我做一个媒体中心软件"

调度器分析：
  难度评估: 中高 → 在线模型
  任务分解:
    1. 需求分析 → 子 Agent A
    2. 数据库设计 → 子 Agent A 或 B
    3. UI 设计 → 子 Agent C
    4. 核心播放 → 子 Agent D
    5. 集成测试 → 子 Agent E
    6. 打包部署 → 子 Agent F

  并行度: 1→2/3/4 可并行 → 5→6
  估计技能: react, sqlite, ffmpeg, pyqt? → 缺 pygame → 触发技能申请
```

**关键设计**：
- 任务 DAG（有向无环图），支持并行和依赖
- 子任务可递归分解
- 进度实时上报
- 支持暂停/恢复/取消
- 任务完成自动通知（跨会话）

### 3.2 技能生态 (Skill Ecosystem) ⭐ 核心创新

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│ 缺口感知 │───→│ 技能申请 │───→│ 自动审核 │───→│ 技能开发 │
│Gap Sense │    │ Request │    │ Review  │    │ Develop │
└─────────┘    └─────────┘    └─────────┘    └────┬────┘
                                                   │
                    ┌──────────────────────────────┘
                    │
              ┌─────▼─────┐    ┌─────────┐
              │ 测试验证   │───→│ 装备上线 │
              │ Validate  │    │ Equip   │
              └───────────┘    └─────────┘
```

**每个环节**：
1. **缺口感知**：执行任务时发现缺少能力 → 记录为技能缺口
2. **技能申请**：生成申请单（为什么需要、预期功能、与现有技能的差异）
3. **自动审核**：空闲时审核申请单，评估必要性（去重、边界检查、安全性）
4. **技能开发**：调用 Agent 开发 SKILL.md + 脚本，写入技能仓库
5. **测试验证**：用样本任务验证技能可用
6. **装备上线**：注册到技能注册表，立即可用

**审核规则**：
- 不与现有技能重叠 >80%
- 非恶意/危险操作
- 有明确的使用场景
- 初版自动通过，后期可引入人工审批

### 3.3 记忆系统 (Memory System)

```
┌─────────────────────────────────────┐
│           记忆写入入口               │
│   (每次交互都写入，不可跳过)          │
└────────────────┬────────────────────┘
                 │
     ┌───────────┴───────────┐
     │                       │
┌────▼─────┐          ┌──────▼──────┐
│ 文件层    │          │ 数据库层     │
│ (原始)   │ ◄────────┤ (高效)      │
│          │  同步    │              │
│ .md 格式 │          │ SQLite/向量  │
│ 人类可读  │          │ 语义检索     │
│ 按日归档  │          │ 关联查询     │
└──────────┘          └─────────────┘
```

**双重设计理由**：
- **文件层**：永远不丢。md 格式，人类可读，Git 友好，备份简单
- **数据库层**：语义检索快到毫秒级。向量嵌入 + 全文索引 + 时间/实体/话题多维度查询

**严格记录规则（铁律）**：
> ⚠️ 每次对话交互后必须写入文件层记忆，不可跳过。文件层是最后兜底，防止数据库损坏/进程崩溃/意外中断导致记忆丢失。

```
每次对话后自动执行：
  1. 写入 memory/YYYY-MM-DD.md（追加）
  2. 同步写入数据库层
  3. 生成短摘要（≤3 条要点）写入会话表
  4. 关键决策/教训/实体同步到长期记忆索引
```

- 对话中禁止仅在内存缓存，必须落盘
- 数据库写入失败 → 降级为纯文件写入，不阻塞主流程
- 文件层不可被删除或覆盖（追加模式）

**存储内容**：
- 决策记录（为什么这么做）
- 教训记录（什么做错了，怎么对的）
- 实体信息（人、项目、公司）
- 对话摘要（每会话一个摘要节点）
- 技能使用统计（哪些技能好用/不好用）

### 3.4 模型适配层 (Model Adapters) ⭐ 核心创新

**磨合期流程**：

```
新模型接入

 ┌─── 阶段一：能力探测 ───┐
 │ 发送一组标准探针         │
 │ - 工具调用支持？         │
 │ - 上下文窗口大小？       │
 │ - JSON 输出能力？        │
 │ - 推理速度？             │
 │ - 多轮对话稳定性？       │
 └──────────┬──────────────┘
            │
 ┌──────────▼──────────────┐
 │ 阶段二：行为学习         │
 │ 持续交互中记录：          │
 │ - 偏好哪种 prompt 格式？  │
 │ - 工具调用频率偏好？      │
 │ - 容易在什么场景崩溃？    │
 │ - 什么 token 数开始退化？ │
 │ - 是否需要 CoT 提示？     │
 └──────────┬──────────────┘
            │
 ┌──────────▼──────────────┐
 │ 阶段三：策略固化         │
 │ 生成针对该模型的交互模板  │
 │ - prompt 包装规则        │
 │ - 工具数量限制           │
 │ - 会话长度阈值           │
 │ - 重试策略               │
 │ - 超时配置               │
 └─────────────────────────┘
```

**智能路由**：

```
任务进来
  ↓
难度评估（快速模型分析）
  ├── 简单（闲聊/查资料）→ 本地模型
  ├── 中等（编码/分析）  → 本地模型（优先）/ 在线（过阈值）
  └── 困难（架构/创作）  → 在线模型
       ↓ 如果在线不可用（欠费/断网）
       └── 兜底：降级到本地模型，加 CoT prompt
```

**成本控制**：
- 在线模型每日/每周预算上限
- 本地模型优先策略（省钱）
- 用量统计和预测

### 3.5 多 Agent 协作

```
主 Agent（永不直接执行危险操作）
  │
  ├── 子 Agent 1：限时运行，独立上下文
  ├── 子 Agent 2：不同模型，不同技能
  └── 子 Agent N：并行执行
       │
       └── 结果汇聚到主 Agent → 综合判断 → 下一步
```

**协作协议**：
- 主 Agent 分配任务 + 上下文片段
- 子 Agent 用独立模型/参数运行
- 子 Agent 间通过消息总线通信（有限制）
- 结果格式统一（JSON），含置信度
- 超时/异常自动回收

### 3.6 审计系统

**全部留痕**：
- 谁（哪个 Agent）在什么时候做了什么
- 调了什么工具、传了什么参数、返回什么
- 技能开发/变更历史
- 模型切换记录和原因
- 错误和恢复过程

**审计后端**：SQLite → 后续可升 PostgreSQL
**审计前端**：Web 面板，时间线视图，关键词搜索

#### 中断与恢复机制

```
用户发起会话
  │
  ├── 检测到后台任务正在执行
  │     │
  │     ├── 完成当前子任务（最多等 30s）
  │     │  或到达安全点
  │     │
  │     ├── 保存检查点 (checkpoint.json)
  │     │   { lastTask: 4, completed: [1,2,3], dagState: {...} }
  │     │
  │     ├── 更新 PROGRESS.md
  │     │   status: paused
  │     │
  │     └── 写入 JOURNAL.md 本次执行日报
  │
  └── 切换到用户会话模式
        │
        ├── 展示当前进度（"项目 X 已完成 35%，上次停在步骤 4"）
        ├── 用户交互...
        └── 会话结束 → 检查待办 → 恢复后台推进
```

#### 优先级系统（三档制）

| 优先级 | 标识 | 触发条件 | 示例 |
|--------|------|----------|------|
| 🔴 紧急 | P0 | 用户明确叮嘱 / 上次中断的任务 | "赶快把这个修了" / 恢复被中断的开发 |
| 🟡 高 | P1 | 技能开发 / 项目推进 / 记忆整理 | 审核技能申请 / 写代码 / 总结记忆 |
| 🟢 中 | P2 | 常规检查 | 邮件、日历、git status |

- 同优先级严格 FIFO
- P0 可抢占 P1/P2
- 不搞复杂权重，简单可预测

#### 活跃项目概念

```
多个项目共存时：
  
  projects/
  ├── agent-system/     ← active: true (当前) → 闲置时推进
  ├── media-center/     ← active: false        → 搁置中
  └── stock-trader/     ← active: false        → 搁置中

规则：
  - 同时只有一个活跃项目
  - 用户说"继续做 XX" → 切换活跃项目
  - 活跃项目暂停超过 N 天 → 自动标记为搁置
```

### 3.8 全局配置系统 (Global Config System)

**设计目标**：所有模块的配置参数统一管理，支持热重载，用户可直接编辑 YAML 文件。

#### 配置层级

```
config/agent-system.yaml  ← 用户编辑
        ↓
  js-yaml 解析
        ↓
  src/config/agent-system-config.ts
    AgentSystemConfig 接口定义
    DEFAULT_CONFIG（TypeScript 默认值）
    mergeConfig() 深度合并：默认值 ⊇ YAML 覆盖
        ↓
  currentConfig（运行时快照）
        ↓  getConfig() / getConfigSection() / getNonsenseConfig()
  各模块读取
```

**设计原则**：
1. TypeScript 接口定义完整类型（`AgentSystemConfig`）
2. `DEFAULT_CONFIG` 兜底所有字段，YAML 可选覆盖
3. 热重载后重新深度合并，不重启服务
4. 每个模块通过 `getConfig()` 或 `getXxxConfig()` 读取配置

#### AgentSystemConfig 接口结构

| 顶层字段 | 来源模块 | 关键属性 |
|----------|---------|---------|
| `agent` | SmartAdapter | callTimeoutMs, maxRetries, emptyLoopThreshold |
| `context` | ContextManager | maxTokens, hotWindowSize, attentionEnabled |
| `nonsense` | NonsenseDetector | checkIntervalMs, thresholds, customRules |
| `diagnostics` | SessionDiagnostics | maxPingFailures |
| `idleTasks` | IdleTaskManager | defaultCooldownMs |
| `circuitBreaker` | CircuitBreaker | failureThreshold, resetTimeoutMs |
| `checkpoint` | Checkpoint | contextWindow, maxRecoveryAttempts |
| `probes` | CapabilityProbe | concurrency, timeoutMs |
| `profiles` | ProfileStore | dataDir |

#### 命令接口

| 子命令 | 功能 |
|--------|------|
| `/config show` | 打印当前完整配置 |
| `/config reload` | 热重载 YAML → 重启 NonsenseDetector 监控 |
| `/config path` | 打印配置文件路径 |

**热重载范围**：
- NonsenseDetector：阈值、自定义规则、崩溃模式即时生效
- 其他模块：配置值已存于 currentConfig，各模块读配置入口需逐一接入

**相关文件**:
- `src/config/agent-system-config.ts` — 接口 + 加载 + 合并 + 格式化
- `src/config/index.ts` — barrel export
- `config/agent-system.yaml` — 可编辑配置文件
- `src/resilience/nonsense-detector.ts` — 配置化重构
- `src/core/agent/agent-core.ts` — `/config` 命令

## 四、技术选型

| 层面 | 选型 | 理由 |
|------|------|------|
| 运行时 | Node.js v22+ | 你已熟悉，npm 生态好 |
| 数据库 | SQLite (better-sqlite3) | 零配置，单文件，足够初期用 |
| 前端 | 纯 HTML/JS SPA | 最小依赖，后续升级 React |
| 向量检索 | better-sqlite3 + 本地嵌入 | 初期用 sqlite-vss 或 faiss-node |
| 语言 | TypeScript | 类型安全，后期维护性好 |
| 配置 | JSON/YAML | 人类可编辑 |
| IPC | 子进程 + stdin/stdout JSON | Agent 间通信 |
